# Scientific Multimodal GraphRAG

Вопросно-ответная система над корпусом научно-технических документов горно-металлургической отрасли: извлекает из файлов факты с обязательной привязкой к источнику, строит граф знаний и отвечает на вопросы естественным языком с проверяемыми сносками.

## Аннотация

Экспертное знание R&D-подразделения рассредоточено по отчётам, статьям, презентациям и таблицам, и поиск конкретного результата — «что известно про отжиг сплава X при 700 °C» — сводится к ручному перебору сотен файлов. Система решает задачу *прослеживаемого* доступа к такому корпусу: каждый документ разбирается на адресуемые фрагменты (страница, слайд, строка таблицы), из фрагментов LLM извлекает структурированные факты, факты валидируются по числам и единицам измерения и проецируются в граф знаний Neo4j. Ответ на вопрос генерируется строго по найденному материалу и сопровождается сносками, механически разрешаемыми в конкретные фрагменты конкретных файлов, — утверждение без источника в ответ не попадает. Семантический индекс строится локальной моделью эмбеддингов bge-m3 (pgvector); извлечение и генерация работают через Yandex AI Studio с каскадом на запасной OpenAI-совместимый LLM для ответов. Достоверность каждого ответа выражена количественно: уровень уверенности и доля научных публикаций среди процитированных документов. Проект разработан в рамках хакатона «Норникеля».

## Архитектура

| Сервис | Порт | Назначение |
| --- | --- | --- |
| **backend** | 8000 | FastAPI: приём документов, конвейер обработки, вопросно-ответный оркестратор; веб-интерфейс «Графа» на `/ui`, OpenAPI на `/docs` |
| **ml-extraction** | 8002 | Извлечение фактов (`/extract`), эмбеддинги bge-m3 (`/embed`), LLM-шлюз (`/chat_json`, `/chat_stream`), веб-поиск (`/web_search`, `/web_answer`) и реестр его площадок (`/web_sources`); Yandex AI Studio, офлайн-перевод RU→EN |
| **ml-mock** | 8001 | Детерминированная заглушка извлечения для автономного режима (базовый compose без override) |
| **PostgreSQL 16 + pgvector** | 5432 (15432 с override) | Персистентность: документы, фрагменты, факты, задачи, векторный индекс |
| **Neo4j 5** | 7474 / 7687 | Граф знаний — проекция утверждений и онтологических сущностей |
| **MinIO** | 9000 / 9001 | Хранилище оригиналов загруженных файлов |

Поток данных: файл поступает в backend, оригинал сохраняется в MinIO, парсер разбирает содержимое на фрагменты; ml-extraction возвращает кандидаты фактов, backend валидирует их и записывает в PostgreSQL с проекцией в Neo4j. Вопрос проходит через оркестратор: поиск по графу и векторному индексу, генерация ответа LLM, перевод цитат в канонические ссылки.

Каталог `./domain` (онтология, синонимы, единицы, правила валидации) монтируется read-only в backend и ml-extraction: изменения применяются перезапуском контейнеров, без пересборки образов. Одноразовый сервис `minio-init` создаёт bucket при первом запуске; порядок старта задан healthcheck-зависимостями — backend запускается только после готовности PostgreSQL, Neo4j и MinIO.

## Технологический стек

- **Python 3.12 · FastAPI · Pydantic v2** — оба сервиса; HTTP между ними — `httpx`.
- **PostgreSQL 16 + pgvector** — документы, фрагменты, факты, задачи, векторный индекс; прямой `psycopg`, без ORM.
- **Neo4j 5 Community (+ APOC)** — граф знаний; официальный Python-драйвер.
- **MinIO** — S3-совместимое хранилище оригиналов и PDF-превью.
- **Парсинг**: `pdfplumber` + `PyMuPDF` (текст и рендер сканов), `python-docx`, `python-pptx`, `openpyxl`; **LibreOffice** (headless) — конвертация DOCX/PPTX в PDF-превью.
- **Эмбеддинги**: локальная `BAAI/bge-m3` (sentence-transformers, 1024 измерения); без неё — детерминированный hash-эмбеддер.
- **LLM**: Yandex AI Studio (OpenAI-совместимый API) с каскадом на запасной OpenAI-совместимый сервер (например, локальный Ollama).
- **Веб-поиск**: `ddgs` параллельно с научными API arXiv, Crossref, Semantic Scholar.
- **Единицы измерения**: собственная таблица нормализации с проверкой размерности (без Pint).
- **Веб-интерфейс**: один HTML-файл без сборки, фреймворков и внешних зависимостей.
- **Docker Compose** — оркестрация всего стека.

## Конвейер обработки

1. **Приём.** `POST /ingest` принимает файл multipart; дубликат распознаётся по SHA-256 до постановки в очередь. Оригинал сохраняется в MinIO, задача передаётся фоновым воркерам (`INGEST_WORKERS`, по умолчанию 2); статус опрашивается через `GET /jobs/{job_id}`.
2. **Разбор на фрагменты.** Парсер разбивает документ на адресуемые единицы: абзацы, строки таблиц, слайды, страницы. Страница PDF считается сканом, если текстовый слой короче 40 символов или растровые изображения покрывают более половины площади, — такая страница рендерится в PNG для визуального извлечения. Для DOCX/DOCM/PPTX LibreOffice строит PDF-превью (хранится в MinIO рядом с оригиналом) — в интерфейсе документ открывается как PDF.
3. **Извлечение фактов.** Фрагменты пакетами отправляются в ml-extraction (`POST /extract`); каждый кандидат факта обязан ссылаться на фрагмент-источник. Отказ LLM не маскируется: документ получает статус `failed` с человекочитаемой причиной.
4. **Валидация.** Числа в факте сверяются с текстом источника в нормализованных единицах (`units.yaml`); правдоподобие значений проверяется по `validation-rules.yaml`.
5. **Автоутверждение и review.** Кандидаты с достаточной уверенностью утверждаются автоматически; остальные ожидают решения эксперта (`/review/*`) и до утверждения в ответах не участвуют.
6. **Проекция в граф.** Утверждённый факт разворачивается в подграф Neo4j: `Claim −[:ABOUT]→ Material`, `Claim −[:BASED_ON]→ Experiment`, `Experiment −[:MEASURES]→ Property / −[:PRODUCED]→ Effect / −[:CONDUCTED_BY]→ Laboratory`, `SourceFragment −[:SUPPORTS]→ Claim`; онтологические сущности присоединяются через `[:MENTIONS]`. Противоположные эффекты по одной паре «материал — свойство» помечаются `conflicting` со взаимными ссылками.
7. **Индексация.** Фрагменты кодируются моделью bge-m3 (1024 измерения) и записываются в pgvector для семантического поиска.
8. **Ответ.** `POST /ask` (или потоковый `/ask/stream`) собирает доказательную базу и генерирует ответ строго по ней — со сносками, метриками достоверности, противоречиями и пробелами.

## Методология ответа

**Маршрутизация.** Вопрос сначала проходит эвристический фильтр релевантности по доменному словарю — вопрос вне предметной области отклоняется сразу, без обращения к LLM, графу и эмбеддингам; эвристика ошибается только в безопасную сторону (цифры, единицы или доменный термин отправляют вопрос в конвейер). Далее выбирается ветка: **fast** — классический RAG по векторному и пословному поиску; **full** — полный конвейер с LLM-разбором вопроса, обходом графа и числовыми условиями. Признак полного конвейера один: цифры, маркеры сравнения или явные фильтры в запросе.

**Сбор доказательной базы.** Оркестратор объединяет прямые факты, результаты семантического поиска и — при отсутствии прямого ответа — смежные факты из графа. Всё найденное собирается в пронумерованный evidence pack; модель видит только его. Независимый веб-контур (`POST /web/answer` → ddgs + arXiv, Crossref, Semantic Scholar) запускается параллельно с каждым вопросом и не смешивается с ответом по базе: его результат отображается за отдельным переключателем в интерфейсе и в базу знаний не записывается. Состав площадок задаётся реестром в карточке «Об этом ответе» (настройки сохраняются в браузере); для англоязычных научных API вопрос переводится локальной моделью с доменным глоссарием, выдача фильтруется по семантической близости к вопросу (bge-m3) — нерелевантные результаты в сводку не попадают.

**Цитирование.** Модель ставит в тексте номерные маркеры `[3]` / `[3, 7]` — по номерам элементов evidence pack; придумывать номера запрещено. Backend механически переводит номера в канонические идентификаторы фрагментов, поэтому фальсификация ссылки невозможна. Списки различаются по смыслу: `sources` — полнота (топ найденного набора), сноски — только реально процитированное; каждый маркер гарантированно разрешается в источник. В интерфейсе сноска открывает документ во встроенном просмотрщике pdf.js на нужной странице с подсветкой цитаты.

**Метрики достоверности.** Ответ несёт `confidence` (0–1), `evidence_status` (`direct` / `partial` / `none`) и `scientific_share` — долю научных публикаций среди различных документов, на которые опёрся ответ. Тип и научность документа определяет LLM по титульным страницам при загрузке — с обоснованием-цитатой, отображаемым в досье документа; происхождение и год определяются эвристиками без LLM, при расхождении приоритет у имени файла. Сбои LLM не скрываются: деградированный ответ содержит `llm_error`.

**Граф ответа.** В поле `graph` ответа (и в интерфейсе) возвращается не сырая проекция Neo4j, а компактный читаемый граф, собранный из фактов ответа: узлы шести типов — `Material`, `Process`, `Equipment`, `Condition`, `Claim` (утверждение, подписанное сутью: «свойство: направление»), `Document`; рёбра — входы к утверждению (`ABOUT`, `USED_IN`, `MENTIONS`) и ссылка утверждения на документ (`CITES`). Значения-заглушки («не указано», `unknown`) узлов не порождают, одинаковые сущности из разных фактов объединяются в один узел, документ — один узел на файл. В развёрнутом виде граф интерактивен: наведение подсвечивает путь вершины от входов до документа, клик фиксирует подсветку.

Гигиена данных: удаление документа каскадно удаляет его фрагменты, утверждения и осиротевшие эксперименты; скрытие документа (`/documents/{id}/visibility`) фильтруется на стороне backend без перестройки графа. При старте backend дополняет недостающие атрибуты ранее загруженных документов — признаки научности и происхождения, год издания, PDF-превью, векторы фрагментов — в фоне, не блокируя запуск.

## Развёртывание

```bash
cp .env.example .env          # задать YANDEX_API_KEY (и при необходимости YANDEX_FOLDER_ID)
docker compose up -d --build
```

Интерфейс — http://localhost:8000/ui, состояние сервисов — `GET /health`. Без `YANDEX_API_KEY` стек запускается, но извлечение отвечает явной ошибкой авторизации (документы получают статус `failed`). Для полностью автономного режима достаточно переименовать `docker-compose.override.yml` — извлечение переключится на `ml-mock`, эмбеддинги — на детерминированный hash-эмбеддер.

Ключевые переменные (`.env`, значения по умолчанию — в `docker-compose*.yml`):

| Переменная | Назначение |
| --- | --- |
| `YANDEX_API_KEY` / `YANDEX_FOLDER_ID` | Ключ Yandex AI Studio; обязателен для реального извлечения |
| `YANDEX_BASE_URL` / `YANDEX_MODEL` | Основной LLM-провайдер (по умолчанию `qwen3.6-35b-a3b/latest`) |
| `FALLBACK_BASE_URL` / `FALLBACK_MODEL` / `FALLBACK_API_KEY` | Запасной OpenAI-совместимый LLM для ответов; пустой URL — ступень отключена |
| `DATABASE_URL` | PostgreSQL + pgvector; без неё backend работает in-memory, без персистентности |
| `NEO4J_URI` / `NEO4J_USER` / `NEO4J_PASSWORD` | Подключение к графу |
| `MINIO_ENDPOINT` / `MINIO_ACCESS_KEY` / `MINIO_SECRET_KEY` / `MINIO_BUCKET` / `MINIO_SECURE` | Хранилище оригиналов и PDF-превью |
| `LLM_CHAT_URL` | Адрес LLM-шлюза для генерации ответов (по умолчанию `http://ml-extraction:8002/chat_json`) |
| `EXTRACTION_SERVICE_URL` / `EXTRACTION_TIMEOUT` | Сервис извлечения; override поднимает таймаут до 600 с |
| `EMBEDDINGS_URL` / `EMBEDDING_DIM` / `EMBEDDING_TIMEOUT` | Провайдер эмбеддингов; bge-m3 = 1024, deterministic-hash = 64 — размерность обязана совпадать с колонкой `vector` |
| `WEB_ANSWER_URL` / `WEB_ANSWER_TIMEOUT` | Контур веб-поиска; пусто — контур отключён |
| `INGEST_WORKERS` | Воркеры очереди загрузок; `0` — синхронный режим |
| `EXTRACTION_BATCH_SIZE` | Размер пачки фрагментов на один вызов извлечения (по умолчанию 8) |
| `CHAT_DEADLINE` / `LLM_TIMEOUT` / `LLM_RETRIES` / `LLM_CONCURRENCY` | Настройка LLM-каскада в ml-extraction: общий бюджет вызова, таймаут, число повторов, параллелизм |
| `WEB_SEARCH_TIMEOUT` / `WEB_LLM_TIMEOUT` | Бюджеты веб-контура: поиск и генерация сводки |
| `WEB_RELEVANCE_MIN` | Порог семантического фильтра веб-выдачи по bge-m3 (по умолчанию 0.45) |

Замечание о резервном провайдере: извлечение при загрузке выполняется **только** через основной; каскад «основной → резервный» действует лишь для генерации ответов. В качестве резервного подходит локальный Ollama (`FALLBACK_BASE_URL=http://host.docker.internal:11434/v1`, ключ не требуется); ключ Яндекса на резервный сервер не отправляется.

## API

Базовый URL — `http://localhost:8000`; полная спецификация — на `/docs`.

| Метод | Путь | Назначение |
| --- | --- | --- |
| POST | `/ask` | Вопрос-ответ: `summary`, факты, источники, граф, противоречия, пробелы, `confidence`, `scientific_share` |
| POST | `/ask/stream` | То же SSE-потоком: `evidence` → `delta`… → `final` |
| POST | `/web/answer` | Независимый веб-контур (UI вызывает его параллельно с каждым вопросом): `{found, answer, url, snippets, llm_error}`; принимает реестр площадок `sources` |
| GET | `/web/sources` | Реестр веб-площадок по умолчанию (прокси `/web_sources` из ml-extraction, кэш на процесс) |
| POST | `/ingest` | Загрузка файла (multipart) → `{job_id, filename, status: "queued"}`; дубликат по SHA-256 возвращается сразу |
| GET | `/jobs/{job_id}` | Статус фоновой задачи: `queued / processing / completed / failed` + `error` |
| GET | `/documents` | Список документов со статусами, признаками `hidden`, `is_scientific`, `doc_type`, `origin`, `year` и счётчиками фактов/экспериментов |
| GET | `/documents/{id}` | Документ и его фрагменты |
| GET | `/documents/{id}/original` | Оригинал из MinIO: PDF — inline, прочие форматы — attachment |
| GET | `/documents/{id}/preview` | PDF-превью DOCX/DOCM/PPTX (LibreOffice), inline; 404 — превью отсутствует |
| POST | `/documents/{id}/visibility` | `{"hidden": true\|false}` — скрыть/показать документ во всех ответах без удаления данных |
| POST | `/documents/{id}/reprocess` | Повторное LLM-извлечение по сохранённым фрагментам и переоценка типа, научности, года; выполняется через общую очередь загрузок |
| DELETE | `/documents/{id}` | Каскадное удаление: фрагменты, факты, векторы, узлы графа, файл в MinIO |
| GET | `/facts/random` | Случайный утверждённый факт из нескрытого документа с LLM-формулировкой «Знаете ли вы?» (`teaser`, `question`) из кэша |
| GET | `/review/facts` | Кандидаты извлечения (опциональный фильтр по статусу) |
| GET | `/review/count` | `{"pending": N}` — кандидаты, ожидающие эксперта |
| POST | `/review/facts/{id}/approve` | Утвердить кандидата; без источника — 400 |
| POST | `/review/facts/{id}/reject` | Отклонить кандидата, опционально `{"note": str}` |
| POST | `/review/facts/bulk` | Пакетно `approve`/`reject` до 1000 id; ошибка по одному id не прерывает пачку |
| GET | `/health` | Состояние PostgreSQL / Neo4j / MinIO / извлечения и LLM-каскада с последними ошибками |

Пример SSE-потока `/ask/stream` — события приходят в строгом порядке, `final` гарантирован даже при сбое конвейера:

```text
event: evidence
data: {"experiments": [...], "sources": [...], "graph": {...},
       "contradictions": [...], "confidence": 0.72, "scientific_share": 0.5, ...}

event: delta
data: {"text": "Отжиг при 700 °C "}

event: delta
data: {"text": "повышает твёрдость [3]."}

event: final
data: {"summary": "...", "experiments": [...], "sources": [...], "graph": {...},
       "contradictions": [...], "gaps": [...], "confidence": 0.72}
```

## Поддерживаемые форматы

| Формат | Обработка |
| --- | --- |
| PDF | Текст постранично (pdfplumber); страницы-сканы рендерятся в PNG (PyMuPDF, 120 dpi) для визуального извлечения; в UI открывается inline |
| DOCX / DOCM | Абзацы и строки таблиц со сквозной нумерацией блоков (python-docx); заголовки стилей становятся разделами фрагментов; PDF-превью через LibreOffice для просмотра в UI |
| PPTX | Текст и таблицы по слайдам (python-pptx); PDF-превью через LibreOffice — в UI презентация открывается как PDF со слайдами |
| XLSX / XLSM, CSV | Построчно, значения связываются с заголовками листа |
| TXT / MD / JSON | Текст блоками; короткие строки присоединяются к содержательным |

Файл с нераспознанным содержимым не теряется: регистрируется фрагмент-заглушка с явной причиной («нужен адаптер, OCR или конвертация»).

## Ограничения

- Полноценного OCR нет: сканы передаются изображениями на визуальное извлечение; `ml-mock` изображения игнорирует.
- Версионирование документов не реализовано: `version_number` всегда 1, повторная загрузка того же файла возвращает существующий документ.
- Рабочее состояние хранится в памяти одного процесса backend; PostgreSQL служит персистентной копией для восстановления при старте, горизонтальное масштабирование ограничено.
- Семантический поиск требует PostgreSQL с pgvector; без `DATABASE_URL` доступен только in-memory-режим без векторного индекса.
- Веб-контур не гарантирует результат: ddgs при капче или ограничении частоты возвращает пустую выдачу (повторы со сменой поисковых движков снижают вероятность, но не исключают её); перевод вопроса для научных API — машинный и на редкой терминологии может ошибаться, глоссарий покрывает лишь известные доменные термины.
- PDF-превью офисных форматов зависит от LibreOffice в образе backend (он заметно увеличивает образ); при сбое или таймауте конвертации превью отсутствует — документ остаётся доступен текстом.
- Ролевой модели доступа и аудита действий нет; CORS открыт для всех источников (включая credentials) — конфигурация для доверенной сети, не для публичного интернета.
- Год издания извлекается как атрибут документа (эвристика по тексту и имени файла), но фильтрация по годам в вопросах не реализована; регион из вопроса участвует в обходе графа.
- Достоверность ответа ограничена корпусом и качеством извлечения: система ускоряет экспертную проверку, а не заменяет её — сомнительные кандидаты проходят через review.
